# Internationalisation (i18n)

Les formulaires sont souvent les composants les plus riches en texte d'une application. Les "placeholders", les messages d'erreur, les étiquettes des champs et les textes des boutons doivent tous être traduits si votre application est multilingue.

InputSpec génère des fichiers `.json` statiques au moment de la compilation. Au lieu de générer un fichier JSON unique pour chaque langue, InputSpec utilise une **approche par squelettes de propriétés** (properties skeleton).

## Comment ça marche

Lors de la compilation de votre projet, le Processeur d'Annotations recherche vos tags `@FieldMeta(displayName="...", description="...")`.

Il crée deux fichiers dans le répertoire `META-INF/difsp` :
1. `[form-id].json` : Le schéma structurel de base.
2. `i18n/[form-id].properties` : Les clés de traduction formatées.

### Exemple de Formulaire

```java
@FormSpec(id = "contact")
public class ContactForm {
    
    @FieldMeta(displayName = "Adresse e-mail", description = "Nous ne partagerons jamais votre adresse e-mail.")
    private String email;
}
```

### Le Squelette de Propriétés Généré

Le compilateur génère `META-INF/difsp/i18n/contact.properties` :

```properties
# Generated i18n properties skeleton for form [contact]

form.contact.title=contact
form.contact.description=

# Field: email
form.contact.field.email.label=Adresse e-mail
form.contact.field.email.hint=Nous ne partagerons jamais votre adresse e-mail.
```

## Utiliser les Propriétés

Ce squelette agit comme votre fichier de base. Vous pouvez maintenant copier ce fichier et le donner à votre équipe de traduction, ou le copier directement dans les bundles `src/main/resources/messages_fr.properties` de votre projet Spring Boot.

```properties
# messages_en.properties
form.contact.title=Contact Us
form.contact.field.email.label=Email Address
form.contact.field.email.hint=We will never share your email.
```

Parce que les clés (`form.contact.field.email.label`) sont générées de manière déterministe à partir de vos noms de propriétés Java, cela garantit que :
1. Vous n'avez jamais de chaînes de traduction orphelines pour des champs qui ont été supprimés de la classe Java.
2. Le frontend (par exemple, React avec `react-i18next`) peut parcourir sans effort le schéma JSON et résoudre `t('form.contact.field.' + currentField.name + '.label')` pour rendre dynamiquement la traduction correcte à la volée !
