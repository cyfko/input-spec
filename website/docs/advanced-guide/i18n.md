# Internationalization (i18n)

Forms are often the most text-heavy components of an application. Placeholders, error messages, field labels, and button texts all need to be translated if your application is multilingual.

InputSpec generates static `.json` files at compile-time. Instead of generating a unique JSON file for every language, InputSpec uses a **skeleton properties approach**.

## How it works

When compiling your project, the Annotation Processor looks for your `@FieldMeta(displayName="...", description="...")` tags.

It creates two files in the `META-INF/difsp` directory:
1. `[form-id].json`: The structural schema.
2. `i18n/[form-id].properties`: Formatted Translation keys.

### Example Form

```java
@FormSpec(id = "contact")
public class ContactForm {
    
    @FieldMeta(displayName = "Email Address", description = "We will never share your email.")
    private String email;
}
```

### The Generated Properties Skeleton

The compiler generates `META-INF/difsp/i18n/contact.properties`:

```properties
# Generated i18n properties skeleton for form [contact]

form.contact.title=contact
form.contact.description=

# Field: email
form.contact.field.email.label=Email Address
form.contact.field.email.hint=We will never share your email.
```

## Using the Properties

This skeleton acts as your English base file. You can now copy this file and give it to your translation team or copy it directly into your Spring Boot `src/main/resources/messages_fr.properties` bundles.

```properties
# messages_fr.properties
form.contact.title=Nous contacter
form.contact.field.email.label=Adresse E-mail
form.contact.field.email.hint=Nous ne partagerons jamais votre e-mail.
```

Because the keys (`form.contact.field.email.label`) are deterministically generated based on your Java property names, it guarantees that:
1. You never have orphaned translation strings for fields that were deleted from the Java class.
2. The frontend (e.g., React `react-i18next`) can effortlessly loop over the JSON schema and resolve `t('form.contact.field.' + currentField.name + '.label')` to render the correct translation on the fly!
